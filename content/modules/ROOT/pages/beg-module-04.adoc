= User-Defined Networks (UDN): IP Persistence and Micro-segmentation

With recent versions of OpenShift, a powerful new feature is available: **User-Defined Networks (UDN)**.

In the previous section, we used a `NetworkAttachmentDefinition` (NAD) to connect to an external corporate network (`corp-network`). This allowed us to maintain a stable IP address during a Live Migration because the IP was managed by an external DHCP.

UDNs also allow for maintaining a unique and persistent IP for our VMs, but with a different approach: they use **OpenShift's own SDN (Software-Defined Network)** instead of depending on an external physical network.

A UDN creates a **flat L2 network** that runs on top of the OpenShift overlay. It acts as a dedicated "network bubble" for a specific set of VMs.

.Key features of a UDN are:

* **Traffic Isolation:** By default, traffic within a UDN is completely isolated. Resources connected to this UDN can only communicate with each other.
* **Integrated IPAM:** The UDN manages its own IP addressing plan (IPAM). It distributes persistent IP addresses to VMs, ensuring that a VM will keep the same IP address even after a migration or restart.

== Creating a User-Defined Network (UDN)
The creation of a UDN is not done via a `NetworkAttachmentDefinition` (NAD), but via the creation of a `UserDefinedNetwork` object that defines the L2 network itself.

For this exercise, **a namespace linked to a UDN has been created for each participant.**

== Micro-segmentation with NetworkPolicies

Now, let's secure our VMs. Our goal:
. Block *all* incoming traffic to `vm-db` by default.
. Allow *only* `vm-web` to connect to `vm-db` on port `3306` (MySQL).
. Block all other flows (such as SSH or ICMP/ping) from `vm-web`.

We do this using standard Kubernetes **`NetworkPolicy`** resources.

# [IMPORTANT]
====
It is crucial to understand that **Network Policies are not specific to UDNs**.

They are standard Kubernetes resources that can be used to control flows on any network managed by OpenShift, including:

.   The default OpenShift SDN (the Pod Network).
.   OVS bridges created by a `NodeNetworkConfigurationPolicy` (NNCP).
.   And, of course, the UDNs we are working with.

. In the left menu, click on *Networking* -> *NetworkPolicies*
. Ensure you are in the correct project (udn-project-X)
. Click on *Create Network Policy*, then *YAML View*

. **Policy 1: Deny All**
+
Create a policy that selects `vm-db` and denies all incoming (ingress) traffic.
+
[source,yaml]
----
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-deny-all
spec:
  podSelector: <1>
    matchLabels:
      vm.kubevirt.io/name: vm-db
  policyTypes:
    - Ingress
  ingress: [] <2>
----
<1> `podSelector` works because the VM is managed by a `virt-launcher` pod that carries our **labels**.
<2> An empty `ingress` list means "allow nothing".
+
. **Policy 2: Allow Web to DB flow**
+
Create a second policy that allows traffic from `vm-web` to `vm-db` *only* on TCP port 3306.
+
[source,yaml]

## apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
name: allow-web-to-db
spec:
podSelector:
matchLabels:
vm.kubevirt.io/name: vm-db <1>
policyTypes:
- Ingress
ingress:
- from:
- podSelector:
matchLabels:
vm.kubevirt.io/name: vm-web <2>
ports:
- protocol: TCP
port: 3306 <3>

<1> This policy applies to `vm-db`.
<2> It allows traffic *coming from* `vm-web`.
<3> It allows traffic *only* on TCP port 3306.

== Verifying Network Flows

Let's test our rules. We will connect to `vm-web` and try to reach `vm-db` (e.g., `192.168.200.12`).

## . Open the `vm-web` console.
. **Test 1: Unauthorized Flow (e.g., PING or SSH)**
+
Try to ping `vm-db`.
+
[source,sh]

## $ ping 192.168.200.12
PING 192.168.200.12 (192.168.200.12) 56(84) bytes of data.
...
--- 192.168.200.12 ping statistics ---
4 packets transmitted, 0 received, 100% packet loss, time 3068ms

*Result:* Failure. ICMP traffic is not allowed by our policy. Attempting `ssh user@192.168.200.12` would also fail.

## . **Test 2: Authorized Flow (TCP/3306)**
+
Use a tool like `telnet` or `nc` (netcat) to test port 3306.
+
[source,sh]

## $ nc -v -z -w 3 192.168.200.12 3306
Connection to 192.168.200.12 3306 port [tcp/mysql] succeeded!

*Result:* Success. The connection is established because it perfectly matches our **`NetworkPolicy`** `allow-web-to-db`.

You have now implemented granular micro-segmentation between your VMs, in addition to providing them with persistent cluster-managed IP addresses, all thanks to UDNs.
