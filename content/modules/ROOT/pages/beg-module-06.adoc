= VM Backup and Restore with Veeam Kasten

== Introduction

In a production environment, the ability to quickly back up and restore virtual machines is crucial. OpenShift Virtualization, being *cloud-native*, integrates with modern data protection solutions like *Veeam Kasten K10*.

Kasten treats applications, including VMs, as an atomic unit, backing up not only the disks (PVCs) but also the configuration (the Kubernetes/OpenShift objects that define the VM).

. In this module, we will:

* Create a test VM and add a file to it.
* Back up this VM using a Kasten policy.
* Simulate a disaster by deleting the VM.
* Restore the VM from the backup.
* Verify that the VM has returned to its previous state, including the test file.

== Preparing the Test Virtual Machine

To test the backup and restore lifecycle, we need a VM with a specific state (a test file).

. Navigate to the *Virtualization* view. Make sure you are in your *project* (`project-X`).

. Click *Create VirtualMachine* and select *From template*.

. Select the *Fedora VM* template.
+
image::beg-module06/1.png[link=self, window=blank, width=100%]

. Change the name to **fedora-kasten-test** and click *Quick create VirtualMachine*.

. Wait for the VM to transition to *Running* status.

. Select the **fedora-kasten-test** VM and click the *Console* tab.

. Log in to the VM using the provided *Guest login credentials*.

## . Once logged in, create an empty test file in the user's home directory root.
+
[source,sh,role=execute]
----
touch test.txt
----

## . Verify that the file exists by listing the directory contents.
+
[source,sh,role=execute]
----
ls
----
+
You should see `test.txt` appear in the list. Leave this VM running.
+
image::beg-module06/2.png[link=self, window=blank, width=100%]

== Accessing the Kasten K10 Console

The Kasten console is the web interface where we will manage backup policies and restore actions.

. In the OpenShift console, switch to the *Administrator* view.

. In the left menu, navigate to *Networking* -> *Routes*.

. Ensure the selected *Project* is **kasten-io** (or the project where Kasten is installed in your lab).

. You should see a route named *k10-route*. Click the URL in the *Location* column.
+
image::beg-module06/3.png[link=self, window=blank, width=100%]

. A new window will open with the Kasten K10 dashboard. You can now log in with your OpenShift credentials.
+
image::beg-module06/4.png[link=self, window=blank, width=50%]

. If necessary, authorize Kasten access for your username.
+
image::beg-module06/9.png[link=self, window=blank, width=100%]

== Creating and Running a Backup Policy
We now have access to the Veeam Kasten console.

We will now create a Policy that tells Kasten what to back up and how.

. Start by clicking the *applications* tile in the dashboard.
+
image::beg-module06/5.png[link=self, window=blank, width=100%]

. In the dashboard, select the three-dot menu for our **project-X** and click *Create a policy*.
+
image::beg-module06/6.png[link=self, window=blank, width=100%]

. Fill in the form for our backup policy:
* *Name:* We can leave this as default.
* *Backup Frequency:* *Daily*.
* Select *Enable Backups via Snapshot Exports* and select the **s3-odf** storage.
* Leave other options as default and click *Submit*.
+
image::beg-module06/10.png[link=self, window=blank, width=70%]

. Click *Create Policy*.

. The policy is now created but has not yet run. On the line of your new policy (`project-x-backup`), click the icon on the right and then *Run Once*.
+
image::beg-module06/11.png[link=self, window=blank, width=100%]

. Confirm by clicking *Yes, continue* in the pop-up window.

. Click *Dashboard* in the main Kasten menu. You will see a backup job in progress (*Running*). Wait a few moments until the job status changes to *Succeeded*.
+
image::beg-module06/12.png[link=self, window=blank, width=100%]

. Our VM is now backed up.

== Simulating a Disaster (Deleting the VM)

It's time to simulate an accidental deletion of our virtual machine.

. Return to the OpenShift console and the *Virtualization* view.

. Make sure you are in your *project* (`project-X`).

. Select the **fedora-kasten-test** VM.

. Click the *Actions* menu (in the top right corner) and select *Stop*, then *Delete*.
+
image::beg-module06/13.png[link=self, window=blank, width=100%]

. Wait for the VM to disappear from the list. Our VM and its `test.txt` file are gone.

== Restoring the Virtual Machine

Now let's use Kasten to restore our VM from the backup point.

. Return to the Kasten K10 dashboard.

. Click *Restore Points* in the menu. You should still see your *project* in the list.

. Click on your *project* name.

. You will see a list of *Restore Points*. Locate the valid restore point we created in the previous step.
+
image::beg-module06/15.png[link=self, window=blank, width=100%]

. Click the *Restore* icon on the same line as this restore point.

. A *Restore Application* window will open. We want to restore everything to its original state. Leave all default options and click the *Restore* button.

. You will be redirected to the *Dashboard* where you can track the restore job status. Wait for the job to change to *Succeeded*.
+
image::beg-module06/16.png[link=self, window=blank, width=100%]

== Verifying the Restore

Let's check if our VM is back and, more importantly, if its state (the `test.txt` file) has been restored.

. Return to the OpenShift console and the *Virtualization* view.

. In your **project-X**, you should now see the **fedora-kasten-test** VM reappear. Wait for it to reach *Running* status.

. Click on the VM, then on the *Console* tab.

. Log in to the virtual machine again (credentials should be the same).

## . Once logged in, list the home directory contents.
+
[source,sh,role=execute]

## ls

**You should see the `test.txt` file!**
+
image::beg-module06/14.png[link=self, window=blank, width=100%]

. (Optional) You can now log out of the console (Ctrl-D).

== Cleanup

To save resources, please stop the VM we just restored.

. In the *Virtualization* view, select the **fedora-kasten-test** VM.
. Click the *Stop* button.

== Conclusion

Congratulations! You have completed the full data protection cycle for a *cloud-native* virtual machine.

We demonstrated how Veeam Kasten integrates with OpenShift Virtualization to back up not only a VM's disk but also its state and configuration. We simulated a disaster by deleting the VM, then successfully restored it to its exact state, including the test file we had created. This shows the power of managing *stateful* applications in OpenShift.
