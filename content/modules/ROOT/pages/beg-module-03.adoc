= Connecting VMs to the Corporate Network

== Introduction

While OpenShift offers a SDN (Software Defined Network) that allows communication between our VMs using the cluster's internal network, some VMs require direct access to our *corporate network*.

This module explains how OpenShift Virtualization bridges the gap between Kubernetes and traditional networks. You will observe the **NodeNetworkConfigurationPolicy (NNCP)** that prepares the nodes, create a **NetworkAttachmentDefinition (NAD)** to define the network, and finally, launch a VM connected to both the pod network and the corporate network.

[[NNCP]]
== Understanding and Observing Node Network Configuration (NNCP)

Before creating a network for our VM, the cluster *nodes* must be physically prepared via the **NodeNetworkConfigurationPolicy (NNCP)**.

An *NNCP* allows for the *declarative* configuration of the physical interfaces of our cluster nodes. For our scenario, an administrator has already created an NNCP that:

1. Selects the worker nodes.
2. Takes a physical network interface (e.g., `enp3s0`).
3. Creates a new **Linux Bridge** (named `br-flat` here) and attaches this physical interface to it.

This `br-flat` bridge is now connected to the corporate network. All VM traffic destined for this network will pass through this bridge. Let's observe this existing configuration.

. In the left navigation menu, click on *Networking* -> *NodeNetworkConfigurationPolicy*.
+
image::beg-module03/23.png[link=self, window=blank, width=50%, height=50%]

. You should see a policy named *br-flat*. Click on it.
+
image::beg-module03/25.png[link=self, window=blank, width=60%, height=50%]

. Click on the *YAML* tab to see the policy definition.
+
image::beg-module03/24.png[link=self, window=blank, width=75%]

. Observe the `desiredState`. It describes the network configuration that OpenShift will apply to the nodes. It will look like this:

[source,yaml]
----
spec:
  desiredState:
    interfaces:
      - bridge:
          options:
            stp:
              enabled: false
          port:
            - name: enp3s0 <1>
        description: Linux bridge with enp3s0 as a port
        ipv4:
          dhcp: false
          enabled: false
        name: br-flat <2>
        state: up
        type: linux-bridge
  nodeSelector:
    node-role.kubernetes.io/worker: '' <3>
----
<1> Defines the port used for the Linux bridge.
<2> Creates the `br-flat` bridge on the node's physical network interface card (NIC) `enp3s0`.
<3> This policy is applied to all nodes with the "worker" role.

== Creating the Network Attachment Definition (NAD)

The NNCP prepared the *nodes*, but it did not create an *attachment* that our VMs can use. For that, we need a **NetworkAttachmentDefinition (NAD)**.

Think of the NAD as a *virtual NIC*. It is a resource created in a *Project* that defines a type of NIC that can be *consumed by our VMs*. It will serve as the link between our *VMs* and our *br-flat Bridge*.

Let's create a NAD now and connect our VMs through our corporate network.

. In the left navigation menu, click on *Networking* -> *NetworkAttachmentDefinitions*.
. In the *Project* dropdown at the top, select the project where you want your VM to reside. We will use the project *project-X*.
. Click the *Create Network Attachment Definition* button.
+
image::beg-module03/26.png[link=self, window=blank, width=100%]
+
. Fill in the form with the following details:
* *Name:* `corp-network`
* *Description:* `Corporate network for VMs`
* *Network Type:* Select `Linux bridge` from the dropdown list. This is the type used by OpenShift Virtualization.
+
. Once the type is selected, the form will update. Fill in the new field:
* *Bridge Name:* `br-flat` (This *must* match the bridge name from the NNCP we observed).
+
NOTE: This is also where you could specify a *VLAN Tag Number* if your network required it. For this exercise, we do not need one and leave this field blank.

image::beg-module03/27.png[link=self, window=blank, width=100%]
+

. Click the *Create* button.
. You will see your new `corp-network` NAD in the list. If you click on it and go to the *YAML* tab, you will see the resulting configuration:

[source,yaml]
----
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: corp-network
  namespace: project-X
spec:
  config: '{
    "cniVersion": "0.3.1",
    "name": "corp-network",
    "type": "bridge", <1>
    "bridge": "br-flat", <2>
    "macspoofchk": true,
    "preserveDefaultVlan": false,
    "ipam": {}
  }'
----
<1> The network plugin type.
<2> The Linux bridge on the node to be used.

[[vm-migration-test]]
== IP Persistence During Live Migration

A key advantage of using a NetworkAttachmentDefinition (NAD) is that network connectivity is managed independently of the underlying pod network. We will prove this by performing a live migration of a VM.

.We expect that:

* The **Pod Network** IP (managed by OpenShift) **will change**, as the VM lands on a new node.
* The **corp-network** IP (managed by our external DHCP) **will not change**.

=== Verify Initial State

. In the *Virtualization* -> *VirtualMachines* list, locate your `fedora-bridged-1` VM.
. Note the IP addresses displayed in the Network section of the Overview.

image::beg-module03/36.png[link=self, window=blank, width=100%]
+

=== Start Live Migration

. Click the action menu at the top right and select *Migration* -> *Compute*.
+
image::beg-module03/38.png[link=self, window=blank, width=100%]
+
. Observe the VM status. It will change to *Migrating*, then return to *Running*. This may take a minute.

=== Verify Post-Migration State

. Once the VM has returned to the *Running* state, check the IP addresses displayed in the Network section of the Overview again.
. Note that the pod network IP address has changed. This is the new Pod Network IP on the new node.
. The IP assigned for the external network has not changed.
+
image::beg-module03/36.png[link=self, window=blank, width=100%]
+

== Migration Conclusion

This step demonstrates a fundamental concept: live migration moved the VM from one physical node to another. The Pod network, being closely tied to the cluster and node infrastructure, had to reassign a new IP address to the VM.

However, the connection to the `corp-network`, defined by the NAD, is a direct connection to the Linux bridge (`br-flat`) that exists on *all* worker nodes. The VM's IP is assigned by a mechanism external to the cluster.

This proves that corporate network connectivity is stable, persistent, and completely independent of the movement of the workload (VM) within the cluster.

[[Résumé]]
== Summary

In this module, you set up hybrid network connectivity for a Virtual Machine. You learned the critical difference between the two components that make this possible:

* **NodeNetworkConfigurationPolicy (NNCP):** The low-level resource, managed by the cluster administrator, that configures the *node's* physical hardware by creating a Linux bridge (`br-flat`) on a physical NIC.
* **NetworkAttachmentDefinition (NAD):** The high-level, namespace-scoped resource that defines a *usable network* by pointing to the NNCP bridge (`br-flat`).
* VMs get their IP addresses from an *external service* (the corporate network's DHCP server).
* VMs can communicate directly with each other *using this external network*.
* These VMs act as true members of that network and could also *reach other devices present on that same network segment*, bypassing the cluster's SDN.
