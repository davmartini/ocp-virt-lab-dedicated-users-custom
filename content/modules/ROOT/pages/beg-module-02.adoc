= Template Management

== Introduction

Templates are the key to standardizing and accelerating virtual machine deployment. They allow you to define a reusable model (operating system, flavor, scripts) to provision new VMs consistently and quickly.

In this part of the workshop, we will explore the different facets of template management:

* We will start by cloning and customizing a Linux template provided by Red Hat to adapt it to a specific need (a database server).
* Next, we will create a template for Windows Server from an ISO image, including automation via `Sysprep`.
* Finally, we will discover *Instance Types*, the new approach introduced in OpenShift 4.14 to simplify VM resource selection.

Let's start by cloning an existing template.

== Cloning and Customizing a Template

By default, the preconfigured templates provided by Red Hat OpenShift Virtualization cannot be customized. However, you can clone a template and make adjustments for your particular *workload* to facilitate the request for specific types of virtual machines for specific *workloads*. In this section of the *lab*, that is exactly what we are going to do, by creating a template that will provide a preconfigured database server on demand for our end users.

== Creating a Windows VM Template

In this part of our *lab*, we will install Microsoft Windows Server 2019 using an ISO hosted on a web server. This represents a way to install an operating system on a virtual machine that takes advantage of the ability to source disks from many locations, including a web server, *object storage*, or other *persistent volumes* in the cluster.

This process can be streamlined after the initial installation of the operating system by creating a cloned root disk from a virtual machine prepared with a *sysprep* script for use with other templates.

. In the left menu, navigate to *Catalog*, and click on the *Template catalog* tab at the top left.

. Type the word *win* in the search bar, or scroll down until you find the *Microsoft Windows Server 2019 VM* tile.
+
image::beg-module02/16_Windows_2k19_Tile.png[link=self, window=blank, width=100%]

. A dialog will appear showing the default configuration linked to the template.
+

+
image::beg-module02/17_Windows_2k19_Dialog.png[link=self, window=blank, width=100%]
+
. In this dialog:

* Specify the name *win-sysprep*
* Check the *Boot from CD* checkbox
* Choose *URL (creates PVC)* from the dropdown menu
* Specify the *image URL*: [https://catalog-item-assets.s3.us-east-2.amazonaws.com/qcow_images/Windows2019.iso](https://catalog-item-assets.s3.us-east-2.amazonaws.com/qcow_images/Windows2019.iso)
* Reduce the CD disk size to *5 GiB*
* Keep the *Disk source* at *Blank* and the size set to the default *60 GiB*
* Ensure that the *Mount Windows drivers disk* checkbox is checked. **This is necessary to install Windows systems, and will provide the *drivers* for *VirtIO*.**

== Introduction to Instance Types

In order to simplify the process of deploying virtual machines, starting from OpenShift 4.14, the default configuration mechanism has been changed to emphasize the use of *Instance Types*. An *instance type* is a reusable object where you can define resources and characteristics to apply to a new VM. You can define custom *instance types* or use the variety included during the installation of OpenShift Virtualization when *provisioning* your own VM. This feels much more like what users experience when using a *self-service* catalog at *cloud providers*.

This section explores *provisioning* a VM using an *instance type*.

. To begin, click on *Catalog* in the left menu. You will see that the default catalog item is *InstanceType*.
+
image::beg-module02/33_Left_Menu_Catalog.png[link=self, window=blank, width=100%]

. The first step when using an *instance type* is to select a volume to boot from. Similar to *templates* that provide *boot sources*, these *boot sources* are available for guests provisioned with an *InstanceType*. You can see the included volumes by selecting the *project* *openshift-virtualization-os-images*, or you can upload your own with the *Add volume* button.
+
image::beg-module02/34_Volume_Boot.png[link=self, window=blank, width=100%]

. Click on the *rhel9* *boot* volume to select it as the type of volume to boot from. The selection will be indicated by a small vertical blue line to the left of the image name and the name itself changing to bold font.
+
image::beg-module02/35_Select_RHEL9.png[link=self, window=blank, width=100%]

. Next, you can select the *instance type* you want to use. There are *instance types* provided by Red Hat by default, or you can create your own for your specific use case. If you hover over a provided *instance type*, you can see a description of the intended use.
+
image::beg-module02/36_Select_InstanceType.png[link=self, window=blank, width=100%]
+

* The *instance types* provided by Red Hat are intended for the following uses:
** *N series*: Designed for network-intensive *DPDK* *workloads*, such as *VNFs*.
** *O series*: Specialized general-purpose *instance type* with preconfigured *memory overcommit*.
** *CX series*: Designed for compute-intensive *workloads* by requesting extra dedicated CPUs for additional *function offload*.
** *U series*: The most general or "universal" *instance type*.
** *M series*: Designed for memory-intensive *workloads*.

. Click on the *U series* tile to see a *dropdown list* of resources defined for general *instance types*. The default option here is *medium: 1 CPUs, 4 GiB Memory*. Select it. Again, the selection will be indicated by a blue line and bold font for the *instance type*.
+
image::beg-module02/37_InstanceType_Resources.png[link=self, window=blank, width=100%]

. The last section to complete when *provisioning* using an *instance type* is similar to the *templates* section. You must provide a name for the virtual machine, and select the *storage class* to use for a storage disk. By default, a name will be generated for the VM, and the default *storage class* will be selected. When you are satisfied, click the *Create VirtualMachine* button.
+
image::beg-module02/38_VM_Details.png[link=self, window=blank, width=100%]

. You will be directed to the overview page of the virtual machine, and see that the VM provisioned using an *instance type* is now operational.
+
image::beg-module02/39_VM_Overview.png[link=self, window=blank, width=100%]

[[cleanup]]
== Cleanup

To save resources for the next *lab*, please stop all the VMs you created in this module.

. Navigate to the *Virtualization* *persona* in the left menu, then click on *Virtual Machines*.
. If any VMs show the *Running* status, highlight the VM in the central tree column, and select the *Stop* button or the option in the *Actions* *dropdown* menu.

All VMs should now be in the *Stopped* state.

== Summary

In this section, we learned how to clone and customize an existing *template* to create one that can be used for specific *workloads* like databases. We also learned how to configure one of the existing Windows *templates* that has no *boot source*, and automate its installation process, so that we can create future deployments easily by cloning the installation *PVC* that was created with this VM. We also introduced how to use *instance types* to further customize our virtual machines for specific *workloads* for a more *cloud*-like experience.
